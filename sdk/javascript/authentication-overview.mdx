## Summary

The CometChat JavaScript SDK provides secure authentication methods to log users into your chat application. This page covers user creation, login with Auth Key (for development), login with Auth Token (for production), logout functionality, and real-time login state monitoring with LoginListener.

## When to Use This

- When implementing user authentication flow in your chat application
- When setting up login/logout functionality after SDK initialization
- When migrating from Auth Key to Auth Token for production deployment
- When you need to monitor login state changes across multiple tabs or devices
- When implementing session management and automatic re-authentication

## Prerequisites

- **SDK**: `@cometchat/chat-sdk-javascript` v4.x
- **State**: CometChat must be initialized via `CometChat.init()` before calling any authentication methods
- **Credentials**: App ID, Region, and Auth Key from [CometChat Dashboard](https://app.cometchat.com)
- **For Production**: Auth Token generated server-side via [CometChat REST API](https://api-explorer.cometchat.com/reference/create-authtoken)

## Step-by-Step

### Step 1: Create User

Before you log in a user, you must add the user to CometChat.

1. **For proof of concept/MVPs**: Create the user using the [CometChat Dashboard](https://app.cometchat.com).
2. **For production apps**: Use the CometChat [Create User API](https://api-explorer.cometchat.com/reference/creates-user) to create the user when your user signs up in your app.

<Note>
We have set up 5 users for testing having UIDs: `cometchat-uid-1`, `cometchat-uid-2`, `cometchat-uid-3`, `cometchat-uid-4` and `cometchat-uid-5`.
</Note>

### Step 2: Check Existing Session

Once initialization is successful, you will need to log the user into CometChat using the `login()` method.

We recommend you call the CometChat login method once your user logs into your app. The `login()` method needs to be called only once.

<Warning>
The CometChat SDK maintains the session of the logged-in user within the SDK. Thus you do not need to call the login method for every session. You can use the `CometChat.getLoggedinUser()` method to check if there is any existing session in the SDK. This method should return the details of the logged-in user. If this method returns null, it implies there is no session present within the SDK and you need to log the user into CometChat.
</Warning>

### Step 3: Login using Auth Key (Development Only)

This straightforward authentication method is ideal for proof-of-concept (POC) development or during the early stages of application development. For production environments, however, we strongly recommend using an [Auth Token](#step-4-login-using-auth-token-production) instead of an Auth Key to ensure enhanced security.

<Tabs>
<Tab title="JavaScript">
```javascript
const UID = "UID";
const AUTH_KEY = "AUTH_KEY";

CometChat.getLoggedinUser().then(
  (user) => {
    if (!user) {
      CometChat.login(UID, AUTH_KEY).then(
        (user) => {
          console.log("Login Successful:", { user });
        },
        (error) => {
          console.log("Login failed with exception:", { error });
        }
      );
    }
  },
  (error) => {
    console.log("Something went wrong", error);
  }
);
```
</Tab>

### Step 4: Login using Auth Token (Production)

This advanced authentication procedure does not use the Auth Key directly in your client code thus ensuring safety.

1. [Create a User](https://api-explorer.cometchat.com/reference/creates-user) via the CometChat API when the user signs up in your app.
2. [Create an Auth Token](https://api-explorer.cometchat.com/reference/create-authtoken) via the CometChat API for the new user and save the token in your database.
3. Load the Auth Token in your client and pass it to the `login()` method.

<Tabs>
<Tab title="JavaScript">
```javascript
const AUTH_TOKEN = "AUTH_TOKEN";

CometChat.getLoggedinUser().then(
  (user) => {
    if (!user) {
      CometChat.login(AUTH_TOKEN).then(
        (user) => {
          console.log("Login Successful:", { user });
        },
        (error) => {
          console.log("Login failed with exception:", { error });
        }
      );
    }
  },
  (error) => {
    console.log("Something went wrong", error);
  }
);
```
</Tab>

## API Details

### Login with Auth Key

#### Method Signature

`CometChat.login(UID: string, authKey: string): Promise<User>`

#### Parameters

| Parameter | Type   | Required | Default | Description                                      |
| --------- | ------ | -------- | ------- | ------------------------------------------------ |
| UID       | string | Yes      | -       | The UID of the user that you would like to login |
| authKey   | string | Yes      | -       | CometChat Auth Key from Dashboard                |

#### Return Value

Returns a `Promise<CometChat.User>` that resolves to the logged-in User object containing user details.

### Login with Auth Token

#### Method Signature

`CometChat.login(authToken: string): Promise<User>`

#### Parameters

| Parameter | Type   | Required | Default | Description                                    |
| --------- | ------ | -------- | ------- | ---------------------------------------------- |
| authToken | string | Yes      | -       | Auth Token of the user you would like to login |

#### Return Value

Returns a `Promise<CometChat.User>` that resolves to the logged-in User object containing user details.

### Get Logged-in User

#### Method Signature

`CometChat.getLoggedinUser(): Promise<User | null>`

#### Return Value

Returns a `Promise<CometChat.User | null>` that resolves to the currently logged-in User object, or `null` if no user is logged in.

### Logout

#### Method Signature

`CometChat.logout(): Promise<Object>`

#### Return Value

Returns a `Promise<Object>` that resolves when logout is successful.

### Errors

| Error Code              | Description                          | Resolution                                                    |
| ----------------------- | ------------------------------------ | ------------------------------------------------------------- |
| ERR_UID_NOT_FOUND       | User with specified UID doesn't exist | Create the user first via Dashboard or REST API               |
| ERR_INVALID_AUTH_KEY    | Invalid Auth Key provided            | Verify Auth Key from CometChat Dashboard                      |
| ERR_INVALID_AUTH_TOKEN  | Invalid or expired Auth Token        | Generate a new Auth Token via REST API                        |
| ERR_NOT_INITIALIZED     | SDK not initialized                  | Call `CometChat.init()` before login                          |
| ERR_ALREADY_LOGGED_IN   | User already logged in               | Call `logout()` first or check with `getLoggedinUser()`       |

## Events & Real-time Behavior

### LoginListener

The LoginListener allows you to monitor login state changes in real-time. This is useful for detecting when a user logs in or out from another tab, device, or when their session expires.

### Registering LoginListener

<Tabs>
<Tab title="JavaScript">
```javascript
const listenerId = "unique-login-listener-id";

CometChat.addLoginListener(
  listenerId,
  new CometChat.LoginListener({
    loginSuccess: (user) => {
      console.log("User logged in:", user);
    },
    logoutSuccess: () => {
      console.log("User logged out");
    }
  })
);
```
</Tab>
<Tab title="TypeScript">
```typescript
const listenerId: string = "unique-login-listener-id";

CometChat.addLoginListener(
  listenerId,
  new CometChat.LoginListener({
    loginSuccess: (user: CometChat.User) => {
      console.log("User logged in:", user);
    },
    logoutSuccess: () => {
      console.log("User logged out");
    }
  })
);
```
</Tab>
</Tabs>

### Removing LoginListener

Always remove listeners when they are no longer needed to prevent memory leaks and duplicate event handling.

### Cleanup in React Components

<Tabs>

```javascript
import { useEffect } from "react";

function AuthComponent() {
  useEffect(() => {

    return () => {
      CometChat.removeLoginListener(listenerId);
```

### Event Ordering

- `loginSuccess` is triggered after a successful `login()` call
- `logoutSuccess` is triggered after a successful `logout()` call
- Events are also triggered when login/logout occurs in another tab or device for the same user

## Edge Cases & Pitfalls

<Warning>
**Pitfall 1: Not checking existing session before login**

Always call `getLoggedinUser()` before attempting to login. If a user is already logged in, calling `login()` again may cause unexpected behavior. The SDK maintains session state, so you only need to login once per session.
</Warning>

<Warning>
**Pitfall 2: Using Auth Key in production**

Auth Key should **never** be used in production client-side code. It can be extracted from your app and used to impersonate any user. Always use Auth Token for production deployments.
</Warning>

<Warning>
**Pitfall 3: Not removing LoginListeners**

Failing to remove LoginListeners when components unmount or pages navigate away can cause memory leaks and duplicate event handling. Always use unique listener IDs and remove listeners in cleanup functions.
</Warning>

<Warning>
**Pitfall 4: Duplicate listener IDs**

Using the same listener ID for multiple listeners will overwrite the previous listener. Always use unique, descriptive listener IDs (e.g., `"chat-screen-login-listener"`, `"header-auth-listener"`).
</Warning>

<Warning>
**Pitfall 5: Calling login before init completes**

Always wait for `CometChat.init()` to resolve before calling `login()`. Attempting to login before initialization completes will result in errors.
</Warning>

## Performance & Scale Notes

- **Session persistence**: The SDK automatically persists login sessions. You don't need to store credentials or re-login on page refresh.
- **Auth Token expiry**: Auth Tokens can be configured to expire. Monitor for `ERR_INVALID_AUTH_TOKEN` errors and refresh tokens as needed.
- **Multiple tabs**: Login state is synchronized across browser tabs. Use LoginListener to react to state changes from other tabs.
- **Listener overhead**: Each registered listener adds minimal overhead. However, avoid registering excessive listeners and always clean up unused ones.

## Security Notes

<Note>
**Auth Key vs Auth Token - Critical Security Guidance**

| Method | Use Case | Security Level |
| ------ | -------- | -------------- |
| **Auth Key** | POC/Development only | ⚠️ Low - Can impersonate any user |
| **Auth Token** | **Production (Recommended)** | ✅ High - User-specific, can expire |

**For production applications, always use Auth Token:**
1. Generate Auth Tokens server-side using the [CometChat REST API](https://api-explorer.cometchat.com/reference/create-authtoken)
2. Never expose your Auth Key or REST API Key in client-side code
3. Store Auth Tokens securely and implement token refresh logic
</Note>

<Warning>
**User Creation in Production**

Never use `CometChat.createUser()` with Auth Key in production client-side code. Create users server-side using the [CometChat REST API](https://api-explorer.cometchat.com/reference/creates-user) to keep your Auth Key secure.
</Warning>

<Note>
**UID Restrictions**

UID can be alphanumeric with underscore and hyphen. Spaces, punctuation, and other special characters are not allowed.
</Note>

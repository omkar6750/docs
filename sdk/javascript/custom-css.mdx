## Summary

Custom CSS allows you to personalize and enhance the user interface of the CometChat call screen by applying your own styles to call components. This guide covers the available CSS classes, how to apply custom styles to different call modes, and best practices for maintaining a consistent and stable UI.

## When to Use This

- Customize the appearance of call buttons (mute, video, end call, screen share)
- Apply brand colors and styling to the call interface
- Modify the layout and appearance of participant video containers
- Style user name labels to match your application's design
- Create a consistent visual experience across different call modes (Default, Tile, Spotlight)

## Prerequisites

1. **Chat SDK initialized**: Ensure `CometChat.init()` has completed successfully. See [Setup SDK](/sdk/javascript/setup-sdk).
2. **Calls SDK initialized**: Ensure `CometChatCalls.init()` has completed. See [Calling Setup](/sdk/javascript/calling-setup).
3. **User logged in**: A user must be logged in via `CometChat.login()`. See [Authentication](/sdk/javascript/authentication-overview).
4. **Active call session**: Custom CSS applies to an active call session. See [Default Calling](/sdk/javascript/default-call) or [Direct Calling](/sdk/javascript/direct-call).
5. **CSS knowledge**: Basic understanding of CSS selectors and properties.

## Step-by-Step

### Step 1: Understand the CSS Class Structure

The Calls SDK provides specific CSS classes for different components of the call UI. These classes are consistent across all call modes (Default, Tile, Spotlight).

**Main Container Classes:**

| CSS Class | Description |
| --------- | ----------- |
| `cc-main-container` | The outermost container wrapping the entire call component |
| `cc-bottom-buttons-container` | Container for all action buttons at the bottom of the call screen |
| `cc-name-label` | User name text container displayed on video tiles |
| `cc-video-container` | Container wrapping each participant's video element |

### Step 2: Style the Bottom Action Buttons

The bottom button bar contains all call action buttons. You can style the container and individual buttons.

**Button Container Classes:**

| CSS Class | Description |
| --------- | ----------- |
| `cc-bottom-buttons-container` | Main container for all bottom buttons |
| `cc-bottom-buttons-icon-container` | Individual button wrapper (applies to all buttons) |

**Individual Button Classes:**

| CSS Class | Description |
| --------- | ----------- |
| `cc-audio-icon-container` | Audio/microphone button (unmuted state) |
| `cc-audio-icon-container-muted` | Audio/microphone button (muted state) |
| `cc-video-icon-container` | Video/camera button (enabled state) |
| `cc-video-icon-container-muted` | Video/camera button (disabled state) |
| `cc-screen-share-icon-container` | Screen share button |
| `cc-switch-video-icon-container` | Switch camera button |
| `cc-end-call-icon-container` | End call button |

## API Details

### Available CSS Classes

#### Container Classes

| CSS Class | Description | Applies To |
| --------- | ----------- | ---------- |
| `cc-main-container` | Outermost container for the call component | All modes |
| `cc-bottom-buttons-container` | Container for action buttons at the bottom | All modes |
| `cc-video-container` | Container for participant video elements | All modes |
| `cc-name-label` | User name text container | All modes |

#### Button Classes

| CSS Class | Description | State |
| --------- | ----------- | ----- |
| `cc-bottom-buttons-icon-container` | Generic button wrapper | All buttons |
| `cc-audio-icon-container` | Audio/microphone button | Unmuted |
| `cc-audio-icon-container-muted` | Audio/microphone button | Muted |
| `cc-video-icon-container` | Video/camera button | Enabled |
| `cc-video-icon-container-muted` | Video/camera button | Disabled |
| `cc-screen-share-icon-container` | Screen share button | - |
| `cc-switch-video-icon-container` | Switch camera button | - |
| `cc-end-call-icon-container` | End call button | - |

### CSS Specificity

When overriding default styles, you may need to use `!important` to ensure your styles take precedence over the SDK's default styles.

## Edge Cases & Pitfalls

<Warning>
**Use Only Documented Classes**: Only apply CSS to the classes specified in this documentation. Using undocumented CSS classes may cause unexpected UI issues or break with SDK updates.
</Warning>

<Warning>
**Avoid Resizing Grid Container**: Do not resize the `cc-main-container` or grid container dimensions. Altering these can negatively impact the grid layout and cause visual distortions across different call modes.
</Warning>

<Warning>
**CSS Specificity Issues**: The SDK applies default styles that may override your custom CSS. Use `!important` when necessary, but use it sparingly to maintain maintainable stylesheets.
</Warning>

<Warning>
**Mode-Specific Styling**: While CSS classes are consistent across modes, the layout differs. Test your custom styles in all call modes (Default, Tile, Spotlight) to ensure they work correctly.
</Warning>

<Warning>
**Dynamic Class Changes**: Button classes change dynamically based on state (e.g., `cc-audio-icon-container` vs `cc-audio-icon-container-muted`). Ensure you style both states for a consistent experience.
</Warning>

## Performance & Scale Notes

- **CSS Complexity**: Keep CSS selectors simple and avoid deeply nested selectors. Complex selectors can impact rendering performance during calls.
- **Avoid Animations on Video Elements**: Heavy CSS animations on video containers can cause performance issues, especially on lower-end devices.
- **Test Across Devices**: Custom styles may render differently across browsers and devices. Test on target platforms before deployment.
- **Minimize !important Usage**: While sometimes necessary, excessive use of `!important` makes styles harder to maintain and debug.

## Security Notes

<Note>
**Auth Key** is acceptable for POC/development only. For production, use **Auth Token** created server-side via the CometChat REST API.
</Note>

<Note>
**CSS Injection**: If allowing user-provided CSS or dynamic styles, sanitize inputs to prevent CSS injection attacks that could alter the UI maliciously.
</Note>

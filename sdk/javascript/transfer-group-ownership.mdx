## Summary

The CometChat JavaScript SDK enables group owners to transfer ownership of their groups to other members. This operation is essential because group owners cannot leave a group without first transferring ownership to another member. The transfer is immediate and grants the new owner full administrative control over the group.

## When to Use This

- When a group owner wants to leave the group but needs to maintain group continuity
- When transitioning leadership responsibilities to another trusted member
- When the current owner is stepping down from their administrative role
- When implementing succession planning for community or team groups
- When the original owner is no longer active and ownership needs to be reassigned

## Prerequisites

- **SDK**: `@cometchat/chat-sdk-javascript` v4.x
- **State**: User must be logged in via `CometChat.login()`
- **Permissions**: User must be the current owner of the group
- **Target User**: The new owner must be an existing member of the group
- **Dashboard**: Ensure your CometChat app is created in the [CometChat Dashboard](https://app.cometchat.com)

## Step-by-Step

### Step 1: Verify Current User is Owner

Before attempting to transfer ownership, verify that the current user is the owner of the group:

<Tabs>
<Tab title="JavaScript">
```javascript

  (group) => {
    console.log("Group retrieved:", group);
    // Check if user is the owner
    if (group.getOwner() === CometChat.getLoggedinUser().getUid()) {
      console.log("User is the owner and can transfer ownership");
    } else {
      console.log("User is not the owner and cannot transfer ownership");
    }
  },
  (error) => {
    console.log("Failed to retrieve group:", error);
  }
);
```

## API Details

### Method Signature

```javascript
CometChat.transferGroupOwnership(GUID: string, UID: string): Promise<string>
```

### Parameters

| Parameter | Type   | Required | Description                                           |
| --------- | ------ | -------- | ----------------------------------------------------- |
| GUID      | string | Yes      | The GUID of the group whose ownership is to be transferred |
| UID       | string | Yes      | The UID of the member who will become the new owner   |

### Return Value

Returns a `Promise<string>` that resolves to a success message when ownership is successfully transferred.

### Errors

| Error Code             | Description                                      | Resolution                                           |
| ---------------------- | ------------------------------------------------ | ---------------------------------------------------- |
| ERR_GROUP_NOT_FOUND    | The specified group does not exist               | Verify the GUID is correct                           |
| ERR_NOT_A_MEMBER       | Target user is not a member of the group         | Ensure the new owner is already a group member       |
| ERR_PERMISSION_DENIED  | Current user is not the owner of the group       | Only the owner can transfer ownership                |
| ERR_USER_NOT_LOGGED_IN | No user is currently logged in                   | Ensure user is logged in before transferring         |
| ERR_INVALID_GUID       | The provided GUID is empty or invalid            | Provide a valid non-empty GUID                       |
| ERR_INVALID_UID        | The provided UID is empty or invalid             | Provide a valid non-empty UID                        |
| ERR_USER_NOT_FOUND     | The target user does not exist                   | Verify the UID is correct                            |

## Events & Real-time Behavior

When ownership is transferred, all group members receive a real-time notification through the group listener. This allows your application to update the UI for all connected users.

### Event Ordering

- The `onOwnershipChanged` event is triggered for all group members when ownership is transferred
- The previous owner's scope automatically changes from `owner` to `admin`
- The new owner's scope changes to `owner`
- Events are delivered in real-time to all connected users
- Offline users will see the updated ownership when they next fetch group details

## Edge Cases & Pitfalls

<Warning>
**Owner Cannot Leave Without Transfer**: A group owner cannot leave a group using `leaveGroup()`. They must first transfer ownership to another member. Attempting to leave as an owner will result in an error. Always check if the user is the owner before showing the leave option.
</Warning>

<Warning>
**Target Must Be Existing Member**: The new owner must already be a member of the group. You cannot transfer ownership to a user who is not a member. Fetch the group members list first and only allow selection from existing members.
</Warning>

<Warning>
**Previous Owner Becomes Admin**: After transferring ownership, the previous owner's scope automatically changes to `admin`, not `participant`. This means they retain administrative privileges but lose owner-specific capabilities like transferring ownership again.
</Warning>

<Warning>
**Single Owner Constraint**: A group can only have one owner at a time. When ownership is transferred, the new owner immediately gains full control, and the previous owner loses owner privileges. There is no co-ownership model.
</Warning>

<Warning>
**Irreversible Without New Owner's Consent**: Once ownership is transferred, the previous owner cannot reclaim it without the new owner's cooperation. The new owner must transfer ownership back. Implement confirmation dialogs to prevent accidental transfers.
</Warning>

## Performance & Scale Notes

- Ownership transfer is a single atomic operation that updates member scopes
- The operation is immediate and does not require additional API calls
- After transfer, refresh group details to ensure your local cache reflects the new ownership
- Consider caching group member lists to provide quick selection of potential new owners

## Security Notes

<Note>
Ownership transfer does not require an Auth Key when performed by a logged-in user who is the current owner. The logged-in user's authentication and owner role are sufficient for authorization.
</Note>

<Warning>
**Validate User Selection**: When accepting a UID for ownership transfer from user input, validate that it corresponds to an actual group member. Never trust client-side data without verification.
</Warning>

<Warning>
**Audit Trail**: Consider logging ownership transfers in your application for audit purposes. This helps track leadership changes and can be important for compliance or dispute resolution.
</Warning>
